# 模型自动更新功能 — 技术方案（修订版 v2）

## 一、背景与目标

当前项目配置了 12 个 LLM 提供商（OpenAI、DeepSeek、通义千问、智谱、Kimi、Claude、文心、MiniMax、阶跃星辰、豆包、Ollama、自定义），各厂商模型频繁迭代更新，旧模型会逐渐下架导致 API 调用失败。

**目标**：每 30 天自动拉取各大厂商最新可用模型列表，检测到变更则自动同步，同时**绝不触碰用户的 API Key**。

---

## 二、现状分析

### 2.1 模型列表散落在 3 处位置

| 位置 | 文件 | 形式 | 问题 |
|------|------|------|------|
| LLM 模块 | `llm/factory.py` — `PROVIDER_PRESETS` | 硬编码 dict | 代码中的权威定义，需手动修改代码 |
| GUI 模块 | `gui/pet_window.py` — `init_default_providers()` | 硬编码 list | 重复定义，与 factory 不同步会出 bug |
| 用户配置 | `llm/config/{provider}.json` | 每个文件存 `model` + `api_key` + `api_base` | 只存当前选中模型，不存模型列表 |

### 2.2 当前数据流

```
main.py
  → load_main_config()
    → 读取 llm_config.json → 获取当前激活的提供商
      → get_full_config(provider_key)
        → PROVIDER_PRESETS[key].models        ← 硬编码的模型列表
        + llm/config/{key}.json               ← api_key 在这里
```

### 2.3 API Key 存储位置

```
llm/config/
├── deepseek.json    → { "api_key": "sk-xxx", "model": "...", ... }
├── openai.json      → { "api_key": "",        "model": "...", ... }
├── doubao.json      → { "api_key": "ark-xxx", "model": "...", ... }
├── ...
└── custom.json
```

## 五、API Key 安全机制

### 5.1 数据隔离

```
                     读取 api_key（只读）
llm/config/{p}.json ──────────────────→ model_updater.py
{ api_key: "sk-xxx" }                        │
                                             │ HTTP 认证用
                                             ▼
                                    GET /v1/models
                                             │
                                             │ 返回模型列表
                                             ▼
                                    model_cache.json    ← 写入模型名（不含 api_key）
                                    { models: [...] }     api_key 字段完全不存在
```

### 5.2 安全保障

- `llm/config/*.json` 的 `api_key` 字段**只被读取，从不被写入**
- 新的 `model_cache.json` 文件中**不包含 api_key** 字段
- 更新日志 `update_log.json` 中**不包含 api_key** 字段
- 即使缓存文件损坏，恢复逻辑从 `factory.py` 的备用硬编码值启动，不碰配置
- 更新操作在独立后台线程执行，不影响 UI 和对话功能

## 七、首次启动兼容（v2 扩展）

| 场景 | 行为 |
|------|------|
| `model_cache.json` 不存在（首次启动） | 使用 `factory.py` 中的内置 fallback 列表，后台静默拉取 |
| 某厂商 API 调用失败 | 跳过该厂商，保留旧列表，记录错误到 update_log.json |
| 某厂商用户未配 API Key | 跳过该厂商，保留旧列表，等用户配置后再更新 |
| 某厂商不支持 `/v1/models`（如百度文心） | 标记 `fetch_status: "unsupported"`，写入跳过理由 |
| 30 天内多次启动 | 不重复拉取，直接使用缓存 |
| 某厂商返回空列表 | 视为异常，保留旧列表，不覆盖为空 |
| Ollama 本地未运行 | 跳过，使用内置 fallback 列表 |

## 八、v2 缺陷修复清单

### 8.1 缺陷 1：更新后内存不刷新（优先级：中）

**问题**：后台 `sync_all_providers()` 写入 `model_cache.json` 后，内存中的 `PROVIDER_PRESETS` 不会自动更新，用户须重启程序才能看到新模型。

**修复方案**：在 `sync_all_providers()` 末尾调用 `_refresh_after_sync()`，执行：
1. 调用 `factory.reload_models()` 刷新 `PROVIDER_PRESETS` 内存值
2. 输出变更摘要到控制台
3. 写入 `update_log.json` 记录本次更新详情

```python
# model_updater.py — sync_all_providers() 末尾新增
def _refresh_after_sync(results: Dict):
    """同步完成后刷新内存中的模型列表"""
    # 延迟导入避免循环依赖
    from .factory import reload_models
    reload_models()
    
    # 记录更新日志
    _write_update_log(results)
    
    # 输出变更摘要
    _print_update_summary(results)
```

### 8.2 缺陷 2：百度文心无法获取模型（优先级：低）

**问题**：千帆 API 不提供标准 `/v1/models` 端点，当前代码会发起请求然后失败。

**修复方案**：在 `fetch_models()` 开头增加已知不支持厂商的预判：

```python
# model_updater.py — fetch_models() 开头新增
UNSUPPORTED_PROVIDERS = {
    "baidu": "千帆 API 不提供标准 /v1/models 端点",
}

def fetch_models(provider_key, api_base, api_key=""):
    if provider_key in UNSUPPORTED_PROVIDERS:
        print(f"跳过 {provider_key}: {UNSUPPORTED_PROVIDERS[provider_key]}")
        return None  # 返回 None 表示不支持，区别于空列表 []
```

同时修改 `sync_provider()` 对 `None` 返回值的处理：
- `None` → 标记 `fetch_status: "unsupported"`，保留旧列表
- `[]`（空列表）→ 视为异常，保留旧列表

### 8.3 缺陷 3：时间周期表述不精确（优先级：低）

**原表述**："每 30 天自动拉取模型列表"

**精确表述**："每 24 小时检查一次 `model_cache.json` 的 `last_update`，若距上次更新 ≥ 720 小时（30 天）则执行全量同步"

实际行为流程：
```
程序启动
  → daemon 线程启动
  → 每 24 小时:
      → is_update_needed()?
        → 检查 model_cache.json 的 last_update 距今是否 ≥ 30 天
        → YES → sync_all_providers() → 写入缓存 + 刷新内存
        → NO  → 跳过，等待下一个 24 小时周期
```

### 8.4 缺陷 4：缺乏更新进度通知（v2 新增）

**问题**：当前为纯后台静默更新，用户完全不知道更新是否发生。

**修复方案**：新增 `update_log.json`，记录每次更新的完整信息：

| 记录字段 | 说明 |
|----------|------|
| `timestamp` | 更新时间 |
| `type` | `auto_update` / `manual_trigger` |
| `summary` | 可读摘要："同步完成: 6 成功, 5 跳过, 1 失败" |
| `details.success` | 成功更新的厂商及其模型变更 (added/removed) |
| `details.skipped` | 跳过的厂商及原因 |
| `details.failed` | 失败的厂商及错误信息 |

日志保留最近 50 条记录，超出时自动清理旧记录。
