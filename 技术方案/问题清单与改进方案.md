## P0 — 线程安全

### 1. 天气获取竞态条件

**文件**: `sensors/environment_sensor.py`
**问题**: `_weather_fetching` 标志位无锁保护，多个线程同时调用 `get_weather_info` 时可能启动多个获取线程，导致重复请求和缓存写入冲突。

**改进方案**:
```python
import threading

class EnvironmentSensor:
    def __init__(self):
        self._weather_lock = threading.Lock()
        # ...

    def get_weather_info(self, city=None):
        # ...
        with self._weather_lock:
            if self._weather_fetching:
                return self._get_cached_weather()
            self._weather_fetching = True
        # 启动获取线程...
```

### 2. 交互决策建议竞态条件

**文件**: `decision/interaction_decider.py`
**问题**: `_pending_suggestion` 的读写没有锁保护，后台总结线程写入与主线程读取 (`pop_pending_suggestion`) 可能产生竞态。

**改进方案**:
```python
class InteractionDecider:
    def __init__(self, ...):
        self._suggestion_lock = threading.Lock()
        self._pending_suggestion = None
        # ...

    def pop_pending_suggestion(self):
        with self._suggestion_lock:
            suggestion = self._pending_suggestion
            self._pending_suggestion = None
            return suggestion

    def generate_behavior_summary(self):
        # ...
        with self._suggestion_lock:
            self._pending_suggestion = suggestion
```

### 4. 声明式规则框架未启用

**文件**: `decision/interaction_decider.py`
**问题**: `SITUATION_RULES` 列表和 `HIGH_PRIORITY_TRIGGERS` 集合已定义但从未在代码中使用。所有规则硬编码在 `_rule_based_decide` 方法中（约 60 行 if-elif），且各规则使用顺序 if 而非 elif，可能导致前面的规则拦截后面的匹配。

**改进方案**:

将硬编码规则迁移至 `SITUATION_RULES` 类变量：

```python
SITUATION_RULES = [
    {
        "name": "long_idle",
        "condition": lambda s: s.get("idle_seconds", 0) > 1800,
        "priority": 3,
        "animation": "wave",
        "message": "主人好久没理我了~",
    },
    {
        "name": "late_night_music",
        "condition": lambda s: s.get("is_late_night") and s.get("music_playing"),
        "priority": 2,
        "animation": "sit_quietly",
        "message": "深夜听歌呢，我安静陪你~",
    },
    # ... 其余规则
]
```

决策方法改为遍历规则列表：
```python
def _rule_based_decide(self, situation):
    matched = []
    for rule in self.SITUATION_RULES:
        if rule["condition"](situation):
            matched.append(rule)
    if not matched:
        return None
    # 按优先级排序，取最高
    best = max(matched, key=lambda r: r["priority"])
    return {"type": "rule", "priority": best["priority"], ...}
```

### 6. MusicTracker 每次操作开关连接

**文件**: `sensors/music_tracker.py`
**问题**: 每次 `record_play`、`get_top_songs` 等方法都执行 `sqlite3.connect()` + `conn.close()`，频繁调用时有不必要的 I/O 开销。`MemorySystem` 已实现 `SQLiteConnectionPool`，但 `MusicTracker` 未复用。

**改进方案**:

方案 A — 引入连接池（推荐）:
```python
class MusicTracker:
    def __init__(self, db_path="music_history.db"):
        self._db_path = db_path
        self._conn = None
        self._lock = threading.Lock()

    def _get_conn(self):
        if self._conn is None:
            self._conn = sqlite3.connect(self._db_path, check_same_thread=False)
            self._conn.execute("PRAGMA journal_mode=WAL")
        return self._conn

    def record_play(self, ...):
        with self._lock:
            conn = self._get_conn()
            # ... 使用 conn

    def cleanup(self):
        if self._conn:
            self._conn.close()
```

方案 B — 复用 `core.memory_system.SQLiteConnectionPool`。

### 8. 音乐播放时长未追踪

**文件**: `sensors/music_tracker.py`
**问题**: `duration` 字段在 `record_play` 中始终写入 0，没有实际的播放时长追踪逻辑。

**改进方案**:

在 `MediaSensor` 中追踪播放起始时间:
```python
class MediaSensor:
    def __init__(self):
        # ...
        self._play_start_time = None
        self._current_track = None

    def _update_music_info(self):
        # ...
        if new_track != self._current_track:
            # 歌曲切换，记录上一首的时长
            if self._current_track and self._play_start_time:
                duration = int(time.time() - self._play_start_time)
                # 通过 callback 传递 duration
            self._current_track = new_track
            self._play_start_time = time.time()
```

### 9. 视频检测仅匹配窗口标题

**文件**: `sensors/media_sensor.py`
**问题**: 视频检测通过窗口标题关键词匹配，无法判断视频是否真正在播放（用户可能只是打开了播放器窗口但暂停/最小化）。

**改进方案**:

短期 — 增加进程状态检查:
```python
def _detect_video(self):
    # 在关键词匹配基础上，检查窗口是否为前台且非最小化
    hwnd = win32gui.GetForegroundWindow()
    is_visible = win32gui.IsWindowVisible(hwnd)
    is_minimized = win32gui.IsIconic(hwnd)
    if is_minimized:
        return  # 最小化窗口不算播放中
```

长期 — 考虑结合 SMTC 或窗口进程状态进行更精确的判断。

## P3 — 架构债务

### 11. add_memory_async 异常静默丢失

**文件**: `core/memory_system.py`
**问题**: `add_memory_async` 使用裸 `threading.Thread` 启动异步写入，线程内无异常捕获，写入失败时静默丢失，无日志无告警。

**改进方案**:
```python
def add_memory_async(self, content, importance=None, metadata=None):
    def _worker():
        try:
            self.add_memory(content, importance=importance, metadata=metadata)
        except Exception as e:
            import logging
            logging.error(f"异步写入记忆失败: {e}", exc_info=True)
            # 可选: 写入失败队列，后续重试

    thread = threading.Thread(target=_worker, daemon=True)
    thread.start()
```

### 12. LLM 重要性评估硬编码模块路径

**文件**: `core/memory_system.py`
**问题**: `_evaluate_importance_with_llm` 方法中直接 `from llm import LLMFactory`，硬编码了模块路径，不利于测试和模块解耦。

**改进方案**:

通过构造函数注入:
```python
class MemorySystem:
    def __init__(self, llm_config=None, llm_factory=None):
        self._llm_factory = llm_factory  # 注入
        # ...

    def _evaluate_importance_with_llm(self, content):
        if not self._llm_factory:
            return 0.5
        provider = self._llm_factory.create(self._llm_config)
        # ...
```

### 14. platform 字段存在重复定义

**文件**: `sensors/media_sensor.py`
**问题**: `_identify_platform` 方法中的平台映射与 `MUSIC_PLAYER_PROCESSES` 类变量存在重复定义，维护时容易不同步。

**改进方案**:

统一为单一数据源:
```python
MUSIC_PLATFORMS = {
    "cloudmusic": {"name": "网易云音乐", "processes": ["cloudmusic.exe", "cloudmusic_mini.exe"]},
    "qqmusic": {"name": "QQ音乐", "processes": ["QQMusic.exe", "QQMusicService.exe"]},
    # ...
}

def _identify_platform(self, process_name):
    for key, info in self.MUSIC_PLATFORMS.items():
        if process_name.lower() in [p.lower() for p in info["processes"]]:
            return info["name"]
    return ""
```
