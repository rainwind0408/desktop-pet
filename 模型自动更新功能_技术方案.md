# 模型自动更新功能 — 技术方案

## 一、背景与目标

当前项目配置了 12 个 LLM 提供商（OpenAI、DeepSeek、通义千问、智谱、Kimi、Claude、文心、MiniMax、阶跃星辰、豆包、Ollama、自定义），各厂商模型频繁迭代更新，旧模型会逐渐下架导致 API 调用失败。

**目标**：每 30 天自动拉取各大厂商最新可用模型列表，检测到变更则自动同步，同时**绝不触碰用户的 API Key**。

---

## 二、现状分析

### 2.1 模型列表散落在 3 处位置

| 位置 | 文件 | 形式 | 问题 |
|------|------|------|------|
| LLM 模块 | `llm/factory.py` — `PROVIDER_PRESETS` | 硬编码 dict | 代码中的权威定义，需手动修改代码 |
| GUI 模块 | `gui/pet_window.py` — `init_default_providers()` | 硬编码 list | 重复定义，与 factory 不同步会出 bug |
| 用户配置 | `llm/config/{provider}.json` | 每个文件存 `model` + `api_key` + `api_base` | 只存当前选中模型，不存模型列表 |

### 2.2 当前数据流

```
main.py
  → load_main_config()
    → 读取 llm_config.json → 获取当前激活的提供商
      → get_full_config(provider_key)
        → PROVIDER_PRESETS[key].models        ← 硬编码的模型列表
        + llm/config/{key}.json               ← api_key 在这里
```

### 2.3 API Key 存储位置

```
llm/config/
├── deepseek.json    → { "api_key": "sk-xxx", "model": "...", ... }
├── openai.json      → { "api_key": "",        "model": "...", ... }
├── doubao.json      → { "api_key": "ark-xxx", "model": "...", ... }
├── ...
└── custom.json
```

## 三、改造方案

### 3.1 文件变更清单

| 文件 | 操作 | 说明 |
|------|------|------|
| `llm/model_updater.py` | **新建** | 模型拉取、对比、同步核心逻辑 |
| `llm/model_cache.json` | **新建** | 上次更新时间 + 模型列表快照 |
| `llm/factory.py` | **修改** | `PROVIDER_PRESETS` 从硬编码改为动态加载 |
| `gui/pet_window.py` | **修改** | `init_default_providers()` 从 factory 读取模型列表 |
| `main.py` | **修改** | 启动时触发后台更新线程 |
| `llm/config/*.json` | **不动** | api_key 存储区，一字不改 |
| `llm/llm_config.json` | **不动** | 只存当前激活提供商 |
| `llm/provider.py` | **不动** | LLM 调用逻辑无关 |

### 3.2 新增模块设计

#### `llm/model_updater.py` — 核心模块

| 类/函数 | 职责 |
|---------|------|
| `fetch_models(provider_key, api_base, api_key?)` | 调用各厂商的 `/v1/models` 接口，返回模型名列表 |
| `compare_models(old_list, new_list)` | 对比新旧列表，返回 `(added, removed)` |
| `sync_all_providers()` | 遍历所有提供商 → fetch → compare → 写入 model_cache.json |
| `is_update_needed()` | 检查 `model_cache.json` 的 `last_update` 是否超过 30 天 |
| `start_background_updater()` | 启动 daemon 线程，按间隔执行 `sync_all_providers()` |

#### `llm/model_cache.json` — 缓存文件

```json
{
  "last_update": "2026-05-18T12:00:00Z",
  "update_interval_hours": 720,
  "providers": {
    "openai": {
      "models": ["gpt-5.5", "gpt-5.5-instant", "gpt-5.5-pro", "gpt-4.1", "o4-mini"],
      "default_model": "gpt-4.1",
      "api_base": "https://api.openai.com/v1",
      "fetched_at": "2026-05-18T12:00:00Z"
    },
    "deepseek": {
      "models": ["deepseek-v4-pro", "deepseek-v4-flash"],
      "default_model": "deepseek-v4-flash",
      "api_base": "https://api.deepseek.com/v1",
      "fetched_at": "2026-05-18T12:00:01Z"
    }
  }
}
```

## 四、各厂商模型获取方式

| 提供商 | 接口 | 认证头 | 需要 API Key | 备注 |
|--------|------|--------|-------------|------|
| OpenAI | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | 标准 OpenAI |
| DeepSeek | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | OpenAI 兼容 |
| 通义千问 (Qwen) | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | DashScope 兼容模式 |
| 智谱 (GLM) | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | 标准接口 |
| Kimi (Moonshot) | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | OpenAI 兼容 |
| 豆包 (ByteDance) | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | Ark API |
| MiniMax | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | OpenAI 兼容 |
| 阶跃星辰 (StepFun) | `GET /v1/models` | `Authorization: Bearer` | ✅ 是 | OpenAI 兼容 |
| 百度文心 (ERNIE) | 千帆专用接口 | `Authorization: Bearer` | ✅ 是 | 非 /v1/models |
| Anthropic Claude | `GET /v1/models` | `x-api-key: xxx` | ✅ 是 | 非 Bearer 方式 |
| Ollama (本地) | `GET /api/tags` | 无需认证 | ❌ 不需要 | 本地接口 |
| 自定义 | — | — | — | 跳过，由用户自行管理 |

> `GET /v1/models` 是只读操作，不消耗 token、不修改配置、不产生费用。

## 五、API Key 安全机制

### 5.1 数据隔离

```
                     读取 api_key（只读）
llm/config/{p}.json ──────────────────→ model_updater.py
{ api_key: "sk-xxx" }                        │
                                             │ HTTP 认证用
                                             ▼
                                    GET /v1/models
                                             │
                                             │ 返回模型列表
                                             ▼
                                    model_cache.json    ← 写入模型名（不含 api_key）
                                    { models: [...] }     api_key 字段完全不存在
```

### 5.2 安全保障

- `llm/config/*.json` 的 `api_key` 字段**只被读取，从不被写入**
- 新的 `model_cache.json` 文件中**不包含 api_key** 字段
- 即使缓存文件损坏，恢复逻辑从 `factory.py` 的备用硬编码值启动，不碰配置
- 更新操作在独立后台线程执行，不影响 UI 和对话功能

## 六、同步流向

```
                  ┌─────────────────────┐
                  │   model_cache.json  │  ← 权威数据源（模型列表唯一真相）
                  └──────────┬──────────┘
                             │ 读取
              ┌──────────────┴──────────────┐
              ▼                             ▼
      llm/factory.py                gui/pet_window.py
      PROVIDER_PRESETS              init_default_providers()
      (动态加载)                     (从 factory 读取)
```

打通后 `pet_window.py` 不再自己硬编码模型列表，改为从 `factory.py` 读取，彻底消除 3 处硬编码不同步的隐患。

## 七、首次启动兼容

| 场景 | 行为 |
|------|------|
| `model_cache.json` 不存在（首次启动） | 使用 `factory.py` 中的内置 fallback 列表，后台静默拉取 |
| 某厂商 API 调用失败 | 跳过该厂商，保留旧列表，日志记录错误 |
| 某厂商用户未配 API Key | 跳过该厂商，保留旧列表，等用户配置后再更新 |
| 30 天内多次启动 | 不重复拉取，直接使用缓存 |
| 某厂商返回空列表 | 视为异常，保留旧列表，不覆盖为空 |
| Ollama 本地未运行 | 跳过，使用内置 fallback 列表 |

## 八、实现步骤

1. **创建 `llm/model_cache.json`** — 空结构或从现有 `PROVIDER_PRESETS` 初始化
2. **创建 `llm/model_updater.py`** — 实现 `fetch_models()`、`compare_models()`、`sync_all_providers()`、`is_update_needed()`、`start_background_updater()`
3. **修改 `llm/factory.py`** — `PROVIDER_PRESETS` 的 models 和 default_model 从 `model_cache.json` 动态加载；新增 `get_models_for(key)` / `reload_models()` 接口
4. **修改 `gui/pet_window.py`** — `init_default_providers()` 改为从 `factory.py` 获取模型列表
5. **修改 `main.py`** — 在模块初始化完成后调用 `start_background_updater()`
6. **测试验证** — 模拟 30 天过期、厂商不可达、无 API Key 等边界情况
